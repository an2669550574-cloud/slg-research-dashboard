# ADR 0006：厂商开发者账号盯新品——出海 SLG 新品的上榜前探测层

- 状态：提议中（2026-07-09，待实现；路由/切片已定，见下）
- 关联：ADR 0005（RSS 早鸟，并列早鸟层）、ADR 0002（新品实机视频）、P1-1 radar 影子行（#190）、
  商店雷达 #176/#208、`docs/PUBLISHERS.md`（厂商主体/雷达账号）、`docs/ARCHITECTURE.md § ST 配额体系`

## 背景 / 问题

现有新品监测全部是**「看榜单」**：movement（tracked 竞品名次异动）、市场/厂商新面孔
（`market_newcomer_log`）、RSS 早鸟（次市场榜维度）——共同前提是**产品已进入被监测的榜单组合**。
但 SLG 几乎都先**软启动 / 封测（CBT）**：不上榜就完全失明。

**实锤（2026-07-09）**：《Hell Asylum: Last Warden》（研发 Rhinos Games 犀牛互动，4399 发行/投资系）
是末日题材 SLG，2026-07 只在 Google Play 封测（`com.us.dyjycbt.google`，cbt 包）。看榜的新品监测
**全链失明**，靠人工看公众号才发现。这类「上榜前」新品正是买量调研最值钱的观察窗，却是当前最大盲区。

## 现状盘点（关键：探测机制已存在，勿重建）

「盯开发者账号」这半**已经实现** = 商店雷达（`itunes_releases.py` / `gp_releases.py`）：

| 环节 | 现状 |
|---|---|
| 探测 | `sync_itunes_releases`（iOS，`publisher_itunes_artists` platform=ios）+ `sync_gp_releases`（GP）；多 storefront 扫、按 track_id diff、幂等、**零 ST** |
| 时机 | scheduler `itunes_releases_sync` = **01:00 UTC**，正好在每日 digest（03:00 UTC）**之前** ✅ |
| SLG 门控 | `entity.is_slg`（纯 SLG 厂 → 新上架全纳入）**或** `is_slg(app_id pin)`（多品类大厂 → 仅钉过的单品） |
| 富化 | SLG 真新上架写 `chart_type='radar'` 影子行 → riding 既有中文化 / subgenre / 🎬 视频管道（`record_radar_newcomers`） |
| 分发 | ① `alert_appstore_releases` 每轮同步后维护者告警；② 每日 digest **仅平淡日 + 仅维护者** 的「🛒 商店雷达近期新上架」折叠段（`build_radar_recent_lines`） |

**三个真缺口**（对应用户方法的落地 delta）：
1. **覆盖薄**：仅接 iOS 64 / GP 32 账号，绝大多数国内出海 SLG 厂商未接账号 → 探测面小（Hell Asylum 正因 Rhinos 未接账号而漏）。
2. **推送弱**：雷达 SLG 新品只在**平淡日**、只发**维护者**、**无 📰 公众号文章**（现只有 🎬 视频）。
3. **核查漏多品类大厂新 SLG**：门控 = `entity.is_slg` OR 已钉 pin；多品类大厂（如 4399，`is_slg=false`）**新出**的 SLG 尚未 pin → 鸡生蛋，抓不到。

## 决策

1. **定位为上榜前早探测层，与 RSS 早鸟（ADR 0005）并列**：两者都是零 ST / 影子行 / 仅维护者的早鸟层，
   互补——RSS 早鸟 = 次市场**榜单维度**补偿（仍要上榜）；账号雷达 = **软启动/封测维度**补偿（不依赖上榜）。
   不替代任何看榜检出，是并列新增。

2. **覆盖是第一杠杆（切片 1，纯数据、零代码、零 ST）**：系统性建档并接入国内出海 SLG 厂商的开发者账号
   （iOS artistId + GP developer id）。机制现成——`/itunes-artist-suggestions` 从已钉/在榜 app 免费反解开发者账号、
   一键接入。**多接一个账号 = 当天就多一份探测，不改任何代码。**

3. **SLG 核查升级（切片 3）**：对已接账号扫出的新 app，除 `entity.is_slg`/pin 门控外，加**主动分类**——
   商店 genre（Strategy）+ LLM `subgenre_cn` 分类，让**多品类大厂新出的 SLG**（现在完全漏）也能浮出，
   并提示「一键 pin」。破鸡生蛋。LLM 走太石网关（已有 #194 成本护栏），仅对新 app 一次性、幂等。

4. **推送升级（切片 2）**：雷达 SLG 新品从「仅平淡日折叠段」升为**有则即显**；补 📰 公众号文章联动
   （复用 `_match_articles_to_apps` / `_newcomer_search_keywords`，雷达影子行现只有 🎬 视频没文章）；
   保留 🎬 视频 + 📝 中文化 + 💻 商店页。

5. **路由 = 仅维护者卡，上榜后再进领导群（尊重减量宪法）**：早鸟未过 ST 口径核实，对领导是噪声风险，
   **不进领导群**。**毕业机制**：早鸟新品真上榜后，由既有看榜检出通道（movement / market-newcomer）
   自然进领导卡——届时影子行已让它带上翻译/视频/文章，领导看到的是「已上榜 + 已富化」的**确认信号**，
   而非未核实早鸟。这与 ADR 0005 RSS 早鸟的分发口径完全一致。

6. **零 ST + 绝不写 game_rankings**：与 radar / RSS 影子行同哲学。开发者账号扫描全走免费源
   （iTunes lookup / GP 页），是本方法能成立的前提（省配额铁律）。影子行 `/history` 排除、不进市场卡片网格。

## 落地切片（纵切、按价值排序，各自可独立验收）

- **切片 1 · 覆盖扩张**（最大价值、风险最低）：产出国内出海 SLG 厂商优先级清单 → 批量建档 + 接开发者账号
  （走 `/itunes-artist-suggestions` 半自动）。**验收**：接入账号数显著上升（iOS 64→目标 150+），
  雷达 `synced` 账号数与影子行检出量随之上升；零 ST、零迁移、可逐个 DELETE 回退。
- **切片 2 · 推送升级**：解「仅平淡日」门控 + 雷达段接 📰 文章 + 保持仅维护者路由。**验收**：
  非平淡日出现新 SLG 时维护者卡也显示，行内带 📝/🎬/📰/💻；领导卡不变。纯代码、可回退旧段。
- **切片 3 · 主动 SLG 核查**：新 app genre + LLM subgenre 分类，多品类大厂新 SLG 浮现 + 提示 pin。
  **验收**：给 4399/网易类账号造一个新 SLG 样本能被识别并提示；开关可关（沿用/新增 flag）。

## 取舍

- **覆盖是持续 curation、非一次性**：先接已知头部出海 SLG 厂商，长尾迭代；可用一次研究扫描产出优先级清单。
- **封测/软启动假阳可接受**：探到但最终没上线/没起量的新品会有——早鸟本就是**线索非结论**，
  不进领导群正是为此消化。
- **跨通道重复可接受**（同 ADR 0005）：早鸟先报维护者、上榜后正常检出进两卡 = 确认信号而非重复噪声；
  `market_newcomer_log` unique 约束按 `chart_type` 隔离，radar/rss/grossing/free 行天然并存。
- **LLM 分类成本**：仅新 app 一次性、幂等，量可控；#194 成本护栏兜底。
- **扫描规模**：现 21 账号 × 5 区 × 3s ≈ 5 min/轮；接到 ~150 账号 ≈ 40 min/轮，日级仍远在
  Apple 礼貌限速红线内（如逼近可拆分账号分批或降扫描区）。

## 回滚

- 切片 1 = 数据：DELETE 开发者账号即退，无代码。
- 切片 2 = 纯代码：回退到「仅平淡日」旧段即可。
- 切片 3 = 开关关闭主动分类。
- **绝不写 game_rankings** 保证零快照污染 → 任何切片回退均无残留。
